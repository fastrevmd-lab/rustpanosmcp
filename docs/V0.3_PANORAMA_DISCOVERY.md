# v0.3 Panorama discovery

This document bounds discovery for Panorama, device groups, templates, and
commit-all. It does not authorize implementation or broaden the standalone
firewall mutation policy.

## Inventory identity

Panorama must be a distinct inventory device type. Discovery must determine
how a caller selects a Panorama appliance, device group, template or template
stack, target serials, and vsys without accepting arbitrary hostnames or XPath
fragments. Stable identifiers must survive display-name changes and must not
allow a token scoped to one device group to address a sibling or parent.

Questions to answer with PAN-OS/Panorama 12.1 lab evidence:

- Which XML API responses provide canonical device-group, template-stack,
  managed-device serial, vsys, and hierarchy identifiers?
- How are shared, parent, and child rulebases represented, and which effective
  scope receives a candidate edit?
- Can inventory reload validate every configured identifier without performing
  a mutation or requiring broad administrator privilege?
- What happens when a managed firewall moves between device groups during an
  approved but unapplied change set?

## Authorization boundary

Bearer grants need separate exact fields for Panorama appliance, device group,
template or stack, target serial set, vsys, action, and permitted XPath root.
Standalone `device` scope must not implicitly grant Panorama. Wildcard tool
scope remains read-only, and approval must bind all target identifiers plus the
ordered action digest and pre-change fingerprints.

Discovery must prove whether inherited/shared configuration requires explicit
parent authorization and how to prevent a narrow child grant from changing a
shared object consumed by other groups.

## Candidate and locking model

Determine whether Panorama exposes locks per administrator, device group,
template, or global candidate and whether configuration and commit locks are
separate. The v0.2.1 invariant remains mandatory: local state may say a lock is
released only after the responsible PAN-OS endpoint confirms removal.

The design must specify serialization keys for:

- edits in the same device group or shared parent;
- templates and template stacks with overlapping managed firewalls;
- concurrent standalone operations against a firewall managed by Panorama;
- commit-all operations whose target serial sets overlap.

## Commit-all lifecycle

Map the exact API sequence and job graph for Panorama commit followed by
commit-all. Required evidence includes queued, running, completed, partially
failed, cancelled, timed-out, and transport-indeterminate outcomes. A single
top-level success is insufficient when individual managed firewalls can fail.

The proposed lifecycle must return and persist:

- Panorama commit job ID and terminal result;
- commit-all parent job plus every target serial's child result;
- candidate and pushed-config fingerprints where the API makes them reliable;
- warnings, excluded/offline targets, and retry eligibility;
- lock ownership/release state and an explicit reconciliation procedure.

No automatic retry is allowed unless the exact target/job state proves the
operation idempotent. Partial success requires a recovery plan; it must never
be represented as a clean global success.

## Approval and audit

Independent approval binds the Panorama identity, device group/template
scope, exact target serials, ordered actions, candidate fingerprints, and an
expiry. Re-resolving a dynamic target group after approval is forbidden; a
membership change invalidates the approval.

Audit events must identify the planner, approver, applier, Panorama, logical
scope, target serial digest, action/XPath fingerprints, all job IDs, final
per-target dispositions, duration, and reconciliation state without logging
payloads or credentials.

## Standalone fallback and compatibility

The current standalone-firewall tools and token schema remain supported.
Panorama support should use new typed inputs or tool names where necessary;
callers must not infer Panorama behavior from a standalone device name.
Discovery must cover a mixed inventory and prove that a Panorama outage does
not disable independent standalone read operations.

## Acceptance gate for an implementation plan

Before v0.3 implementation is planned, capture reproducible lab responses for
inventory hierarchy, candidate changes, locks, Panorama commit, commit-all,
and per-target failure. Resolve every authorization and serialization question
above, document the minimum PAN-OS administrator roles, and write a rollback
procedure for partial commit-all. Then scope multi-vsys and HA-aware discovery
as the following slice rather than silently folding it into Panorama.
