# v0.2.1 Maintenance Design

## Goal

Ship a narrow v0.2.1 patch that makes PAN-OS configuration-lock state
truthful after commit/discard, replaces the lab's private-CA listener
certificate with default-trusted TLS, and proves the result through CI and a
reversible PAN-OS lab rollout.

## Selected design

### Lock reconciliation

`release_config_lock` will return `Result<()>`; it will no longer turn a
device/API failure into a successful local state transition. A successful
commit or discard remains non-terminal until lock removal succeeds. Confirmed
removal clears `config_lock_held` and persists `committed` or `discarded` in
one final state update.

If PAN-OS accepts the commit or revert but lock removal fails, persist the
operation as `indeterminate`, retain `config_lock_held: true`, store recovery
details identifying the completed device action and failed unlock, return an
error to the MCP caller, and leave endpoint serialization blocked. The
existing offline reconciliation command is the explicit operator path after
the live PAN-OS job, candidate, and lock are checked.

The alternatives were optimistic clearing, which produces false unlock state,
and a new compound terminal-state enum, which expands the public lifecycle for
a patch release. `indeterminate` reuses the existing recovery contract and is
the smallest safe representation.

### TLS and credential handling

The lab hostname has no public A record and resolves privately to LXC 608, so
certificate issuance will use Let's Encrypt production ACME with Cloudflare
DNS-01. The issued full chain and private key will remain root/service-readable
on LXC 608 and the native Rust TLS listener will continue on port 30031.

Acceptance requires default system verification from the workstation, Codex,
Claude Code, and a direct TLS probe without a private CA argument. Renewal
will be automated from the existing protected DNS plugin or an equally
restricted renewal hook. Any DNS credential exposed during rollout is rotated
before completion.

### Lab tokens

Rotate only `lab-change-writer` and `lab-change-reviewer`, preserve their
existing least-privilege device/tool/action/XPath grants, update both Codex and
Claude Code protected client configurations, reload the service, prove the old
tokens no longer authenticate, and prove the new tokens do. Plaintext transfer
artifacts are deleted after installation; the server retains only digests.

### Release and rollout

Run the repository's complete formatting, Clippy, test, fuzz-build,
packaging, and release gates. Then run an approved multi-action lab change
through apply, diff, full validation, commit, and a separately approved delete
through validation and commit. The ending candidate fingerprint and PAN-OS
config-lock list must match the baseline.

Commit and push to `main`, require green main CI, tag and publish v0.2.1,
build the release image, snapshot LXC 608, deploy the verified v0.2.1 artifact,
and repeat service/TLS/read-only checks. Preserve the v0.2.0 binary and
snapshot as rollback points.

## v0.3 handoff

After v0.2.1 is proven, create a discovery document—not implementation—for
Panorama inventory, device-group scoping, template boundaries, commit-all job
reconciliation, independent approval, and the standalone-firewall fallback.
Multi-vsys and HA-aware workflows remain the next discovery slice after
Panorama.

## Test matrix

- discard plus successful unlock persists `discarded` and
  `config_lock_held: false`, including after state-file restart;
- successful commit plus successful unlock persists `committed` and false,
  including after restart;
- discard plus failed unlock returns an error and persists `indeterminate`,
  true, and actionable recovery details;
- successful commit plus failed unlock returns an error and persists the same
  safe recovery state while retaining its job identifier;
- existing failed-commit behavior retains the lock for a later discard;
- full workspace tests and the reversible PAN-OS lab acceptance suite pass.
