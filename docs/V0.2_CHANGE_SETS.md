# v0.2 change sets and independent approval

v0.2 adds an approval boundary above the existing PAN-OS candidate lifecycle.
It does not expand the inventory write roots. A request must be allowed by both
the device mutation policy and the writer token's narrower mutation grant.

## Separate tokens

Use different bearer-token names and deliver them to different people or
systems. A writer token needs the planning/apply and normal lifecycle tools,
an explicit mutation grant, and a short expiry:

```bash
rust-panosmcp --device-mapping /etc/rust-panosmcp/devices.json token add \
  --tokens-file /var/lib/rust-panosmcp/tokens.json \
  --name change-writer --devices panosvm \
  --tools get_candidate_fingerprint,create_panos_change_set,get_panos_change_set,apply_panos_change_set,diff_panos_candidate,validate_panos_candidate,commit_panos_candidate,discard_panos_candidate,get_panos_operation \
  --mutation-root "/config/devices/entry[@name='localhost.localdomain']/vsys/entry[@name='vsys1']" \
  --mutation-actions set,delete --expires-in-secs 3600
```

The reviewer token does not receive a mutation grant or apply/commit tools:

```bash
rust-panosmcp --device-mapping /etc/rust-panosmcp/devices.json token add \
  --tokens-file /var/lib/rust-panosmcp/tokens.json \
  --name change-reviewer --devices panosvm \
  --tools get_panos_change_set,approve_panos_change_set \
  --expires-in-secs 3600
```

Wildcard tool scope remains read-only. Expired tokens fail authentication.
Token-store version 1 is accepted and is written as version 2 on the next
update; existing v0.1 tokens have no v0.2 mutation grant.

The v0.1 `stage_panos_config` tool remains available for compatibility and does
not use the approval layer. Before making v0.2 approval mandatory, revoke every
old writer token and issue replacements that omit `stage_panos_config`. Do not
give either the reviewer or ordinary MCP clients the legacy stage tool.

## Required flow

1. The writer gets the candidate fingerprint and calls
   `create_panos_change_set` with 1-64 ordered set/delete actions.
2. The server validates all XML, delete confirmations, inventory roots, and
   token roots/actions, then persists the plan without changing PAN-OS.
3. The reviewer calls `get_panos_change_set`, reviews the exact XML/actions and
   pre-change fingerprint, then calls `approve_panos_change_set` with the exact
   digest. Self-approval and digest mismatch are refused.
4. Within 15 minutes, the original writer calls `apply_panos_change_set` with
   the same digest and fingerprint. The server rechecks current token policy,
   obtains one canonical-endpoint mutex and optional PAN-OS config lock,
   rechecks candidate drift, and applies the actions in order.
5. If any later action fails, the server immediately requests an admin-scoped
   partial revert. A failed revert becomes `indeterminate` and keeps the
   endpoint blocked for manual reconciliation.
6. A successful apply returns a normal operation ID and candidate fingerprint.
   Continue with `diff_panos_candidate` → `validate_panos_candidate` →
   `commit_panos_candidate`, or call `discard_panos_candidate`.

The digest covers the writer identity, device, initial candidate fingerprint,
and ordered action objects. Editing any bound field requires a new plan and
approval. Approval expires 15 minutes after planning and can be used only once.

## Persistence and recovery

Run the server with an absolute private state path:

```text
--state-file /var/lib/rust-panosmcp/mutation-state.json
```

The parent directory must already exist. The file is atomically replaced with
mode 0600 and is rejected if it is a symlink, non-regular file, over 8 MiB, or
group/other-readable. Because it contains exact planned XML elements, protect
and back it up like sensitive candidate configuration.

On restart, planned and approved records remain available. Any operation that
was staging, validating, or committing is changed to `indeterminate`. Active,
failed, or indeterminate records block every inventory alias resolving to the
same canonical PAN-OS endpoint. Reconcile PAN-OS jobs, candidate state, and
configuration locks manually. Then, with the server stopped, explicitly close
the persisted record (the long confirmation is intentional):

```bash
rust-panosmcp state resolve \
  --state-file /var/lib/rust-panosmcp/mutation-state.json \
  --operation-id OPERATION_ID --disposition discarded \
  --confirmation "RESOLVED OPERATION_ID AS DISCARDED"
```

Only `indeterminate` records can be resolved this way. The command does not
contact PAN-OS or remove a lock; it records an outcome you have already proved
from PAN-OS jobs, candidate/running state, admin attribution, and lock state.
Use `--disposition committed` and `AS COMMITTED` only when that outcome is
equally proven.

For production, make `--state-file` mandatory in service management even
though the CLI leaves it optional for v0.1 compatibility and local tests.
